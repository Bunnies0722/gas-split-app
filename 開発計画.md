# 旅行ガソリン代割り勘アプリ 開発計画

## 0. 前提の補足（重要）

依頼文で「e-Stat API」と書かれていましたが、調べた結果、資源エネルギー庁が週次で発表しているレギュラー/ハイオク/軽油の小売価格（[石油製品価格調査](https://www.enecho.meti.go.jp/statistics/petroleum_and_lpgas/pl007/results.html)）は **e-Statには載っていません**。e-Stat上にあるのは別の統計（月次の石油製品需給動態統計）です。

実際のガソリン価格データは、enecho（資源エネルギー庁）のページから毎週水曜14時に更新される **Excelファイル**（ファイル名に日付が入る、例: `260610.xlsx`）として配布されているだけで、REST APIはありません。そのため取得方法は「ページをスクレイピングして最新のExcelリンクを見つけ、ダウンロードして値を読む」というバッチ処理になります。仕様にある「手入力で単価を上書きできるUI」は、この取得失敗時のフォールバックとしても重要な役割を持ちます。

## 1. 技術スタック提案

ヒアリングの結果（Web公開・友達と共有・履歴保存が必要・JS/TS中心）を踏まえた構成です。

| 領域 | 採用技術 | 理由 |
|---|---|---|
| フロントエンド | Next.js（React + TypeScript） | フロント/バックを1つのリポジトリで書け、無料デプロイ先（Vercel）との相性が良い |
| バックエンド | Next.js の API Routes（Node.js） | 別サーバーを立てずに済み、構成がシンプル |
| データベース | Supabase（PostgreSQL、無料プラン） | 無料枠が大きく、TypeScriptとの連携が容易。履歴・グループ保存に使う |
| 距離計算API | OpenRouteService（ORS） | クレジットカード登録不要で無料利用可（1日2,500件）。地名→緯度経度のジオコーディングとルート距離計算の両方を1つのAPIでカバーできる |
| ガソリン価格データ | enecho公式ページのExcelを週次スクレイピング | 公式の正データに最も近い。APIが無いため自前で定期取得する設計が必要 |
| ホスティング | Vercel（無料Hobbyプラン） | フロント・API Routesをまとめて無料でホスティングでき、Cron機能で週次バッチも組める |

備考：距離計算は将来的に精度や安定性を求める場合、Mapbox（月10万件まで無料）への切り替えも検討余地があります。地名はそのまま厳密なジオコーディングAPIに渡すと失敗しやすいため、施設名検索（Places的な検索）を経由して緯度経度を取得する設計にします。

## 2. フェーズ分けロードマップ

挫折しないよう、**毎フェーズで「動くもの」が増える**順序にしています。

### Phase 0: 環境準備
- Node.js, Git, GitHubアカウント, Vercelアカウント, Supabaseアカウントを用意
- ORSの無料APIキーを取得

### Phase 1: 計算ロジック＆UIプロトタイプ（今回作成）
- 外部API・DBなしの単一HTMLファイルで、入力→計算→結果表示→コピー機能までを先に動かす
- ここで計算式・UIの使い勝手を固める

### Phase 2: Next.js化＋距離計算API連携
- プロトタイプをNext.jsに移植
- 出発地・到着地（地名）をAPI Routes経由でORSに渡し、距離(km)を自動計算
- APIキーはフロントに出さず、必ずバックエンド（API Routes）経由で呼ぶ

### Phase 3: ガソリン価格自動取得
- Vercel CronなどでBatch処理を週次実行し、enechoのExcelを取得・解析して最新単価を保存
- 取得失敗時は前回値 or デフォルト値にフォールバック
- 画面側は「自動取得値を表示しつつ、手入力で上書き可能」なUIにする（仕様通り）

### Phase 4: DB導入・履歴/グループ保存
- Supabaseに「旅行（trip）」「グループ（group）」「精算結果」のテーブルを作成
- 過去の精算履歴の一覧・再表示機能を追加

### Phase 5: デプロイ・公開
- Vercelで公開、環境変数（APIキー、DB接続情報）を設定
- 友達に共有するURLを発行

### Phase 6（任意・将来）
- PWA対応（スマホのホーム画面に追加できるように）
- 認証機能（グループごとにログイン分け）
- スマホネイティブ化したい場合はここでSwift/React Nativeを検討

## 3. 計算ロジック（仕様の再確認）

```
走行距離(km) ÷ 燃費(km/L) = 推定使用ガソリン量(L)
推定使用ガソリン量(L) × ガソリン単価(円/L) = 合計ガソリン代(円)
合計ガソリン代(円) ÷ 人数 = 1人あたりの精算金額(円)
```

一度は「給油量(L)をそのまま入力する方式」に変更しましたが、旅行中に給油しなかった場合は給油量自体が分からず計算できないという問題が判明したため、再度「走行距離 ÷ 燃費」で推定する方式に戻しました。ただし燃費の入力を毎回必須にすると手間なので、**目安として15km/L をデフォルト値で入力済み**にしており、自分の車の燃費が分かれば書き換えられる、というバランスにしています。単価は資源エネルギー庁の最新データ（出発地・到着地の地域平均）を自動で提案しつつ、手入力で上書きも可能です。

## 4. ガソリン価格の自動取得について（追記）

「経済産業省（資源エネルギー庁）から自動取得できないか」という点を実際に検証しました。結論：**自動取得は可能ですが、ブラウザのJavaScriptから直接は不可能で、サーバー側での処理が必要**です。

- 資源エネルギー庁のサイトはCORS（外部サイトからのJavaScript経由アクセスを許可する仕組み）に対応していないため、フロントエンドのコードから直接 `fetch()` するとブロックされます。
- そこで `scripts/fetch-gas-price.js` というNode.jsスクリプトを用意しました。これはサーバー側（自分のPCや、将来のVercelのサーバーレス関数）で実行し、結果ページを読んで最新のExcelリンクを見つけ、ダウンロード・解析して `prototype/price.json` に保存します。
- プロトタイプのHTMLはこの `price.json` を読み込んで単価欄に自動入力します。`price.json` が無い・読み込めない場合は手入力にフォールバックするので、アプリは止まりません。
- 注意：Excelの実際のレイアウト（見出しの表記など）は私の実行環境からは確認できなかったため、`fetch-gas-price.js` 内のロジックは一般的な統計表のレイアウトを想定したものです。`npm run fetch:debug` でシートの中身を確認し、必要に応じて調整してください（詳細は `scripts/README.md` を参照）。
- また、`price.json` をローカルで読み込むには `index.html` をダブルクリックで開くのではなく、簡易サーバー経由で開く必要があります（`file://` だとブラウザの仕様でfetchがブロックされるため）。例：`npx serve gas-split-app/prototype` または `python3 -m http.server`。Phase 5でVercelにデプロイすれば、この制約は自然に解消します。
- 価格データの自動更新は、Cowork（このAIアシスタント）のスケジュール実行機能を使って**毎週水曜15時に自動実行**するよう設定済みです（資源エネルギー庁の公表が毎週水曜14時頃のため、その後に実行）。

### 4.1 地域別の平均価格（追記）

全国平均だけだと地域差（地方ごとのガソリン価格の違い）が反映されないという指摘を受け、出発地点・到着地点それぞれの都道府県から資源エネルギー庁の9地域区分（北海道/東北/関東/中部/近畿/中国/四国/九州/沖縄＝経済産業局の管轄と同じ）を求め、**2地点の地域平均価格**を単価欄に提案する仕組みを追加しました。

- `scripts/fetch-gas-price.js` が、すでに解析していた「公表資料（経済産業局別）」シートから全国平均だけでなく9地域分の価格も抽出し、`price.json` に `regionPrices` として保存します。
- `prototype/index.html` 側では、地図のピンを置いたときにNominatim（逆ジオコーディング）から都道府県名を取得し、47都道府県→9地域のマッピング表で地域を特定します。
- 出発地・到着地の地域が分かればその2地域の平均、片方しか分からない場合はその1地域、どちらも分からない場合（海外や未設定など）は全国平均にフォールバックします。
- どの場合も、単価欄に既に値が入っている（手入力済み）場合は上書きしません。
